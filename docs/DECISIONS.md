# 架构决策记录

这份记录用于解释当前版本的关键取舍，方便面试时说明为什么项目这样设计，而不是只展示功能结果。

## 1. 页面层使用 Jinja2 + 本地 CSS + 原生增强脚本

初始计划倾向于 Jinja2 + HTMX + Tailwind CSS，目标是降低复杂前端学习成本。当前版本最终选择 Jinja2 服务端渲染、本地 CSS 和极少量原生增强脚本，原因是：

- 面试演示优先级是稳定闭环：创建、生成、复核、确认、签收、审计和导出必须可靠。
- 页面不依赖外部 CDN，现场断网或平台网络波动时仍能展示核心流程。
- 避免引入 Node/Tailwind 构建链，降低 Windows 本地、Linux CI 和云平台部署差异。
- 当前交互以表单提交、状态展示和导出为主，原生增强脚本已经覆盖加载态等必要体验。

这个取舍不是放弃前端工程化，而是第一版把复杂度集中在业务建模、LLM 适配、失败兜底、审计、版本记录和交付文档上。后续如果要增强局部无刷新体验，可以在不改数据模型和 API 的前提下逐步引入 HTMX。

## 2. 离线兜底是产品能力，不是测试替身

系统在未配置 `LLM_API_KEY`、模型超时、HTTP 错误或结构化解析失败时，会记录审计日志并使用 DB2 场景化合成结果。这样设计是为了保证公开演示和面试现场不被模型额度、网络或供应商稳定性影响。

离线兜底仍然按场景生成风险、Runbook、回滚、前置检查、验收和沟通摘要，因此可以展示数据库运维判断，而不是返回固定占位文本。

## 3. SQLite 本地兜底，PostgreSQL 用于部署

本地默认使用 SQLite，是为了让项目开箱即跑。部署时通过 `DATABASE_URL` 切换到 PostgreSQL，连接串使用 `postgresql+psycopg://`。

当前应用启动时会自动创建表并写入合成案例，同时保留 Alembic 配置。第一版这样做可以降低面试演示门槛；如果进入团队生产化，应改为显式迁移流程，并把启动期建表限制在初始化任务中。

## 4. 交付物采用版本快照而不是只保存最终文本

`artifact_revisions` 会记录 AI 生成、人工编辑、人工确认等事件。这个设计让项目能解释“AI 产出如何被人复核并形成交付结论”，也能支撑版本差异、导出审计和面试追问。

## 5. 公开交付以脚本门禁收口

项目提供本地验收、发布就绪、线上验收、README 链接回填、公开交付审计和交付状态汇总脚本。这样做的目的不是堆脚本，而是让面试交付从“我本地能跑”变成“可以被命令复核”。

严格公开交付仍需要两个外部事实：真实 HTTPS 演示地址和公开视频链接。拿到这两个链接后，`scripts/delivery_status.ps1 -Strict` 是最终状态门禁。
