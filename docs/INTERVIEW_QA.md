# 面试答辩材料

这份材料用于面试现场的技术追问。演示时不需要逐字念，重点是能把产品边界、工程取舍和风险控制讲清楚。

## 30 秒项目介绍

DBA ChangeOps AI 工作台是一套面向数据库运维变更的 AI 交付系统。它把变更需求、SQL、表结构和约束条件，转成风险评估、执行 Runbook、回滚方案、前置检查 SQL、验收清单和沟通摘要。系统不是聊天页面，而是有案例、分析运行、交付物、版本、签收、LLM 审计和 Markdown/PDF 导出的完整闭环。

## 为什么不是聊天机器人

DBA 变更需要稳定交付物和责任边界。纯聊天很难保证每次都有风险、回滚、验收、沟通这些固定材料，也不方便做版本记录、人工确认、签收和导出。这个项目把 AI 输出约束成固定结构，再进入人工复核和审计流程，更接近真实变更评审系统。

## 为什么选择 FastAPI + Jinja2

这个项目的重点是 AI 工作流和交付闭环，不是复杂前端工程。FastAPI 可以同时提供页面和 API，Jinja2 服务端渲染能降低构建成本，配合本地 CSS 和原生增强脚本，现场演示也不依赖外部 CDN。最初考虑过 HTMX/Tailwind，当前版本把这个取舍记录在 `docs/DECISIONS.md`：第一版优先保证业务闭环、部署一致性和离线演示稳定性，后续可以在现有 API 上逐步增强局部无刷新体验。

## 数据模型怎么设计

核心表分成五类：

- `cases`：保存变更输入资料和交付封面元数据。
- `analysis_runs`：保存每次 AI 分析运行、模型状态和交付包签收。
- `artifacts`：保存 6 类当前交付物。
- `artifact_revisions`：保存 AI 生成、人工编辑、人工确认的版本历史。
- `llm_call_logs`：保存模型调用状态、耗时、请求响应和失败原因。

这样拆分的好处是页面、API、导出和审计都能围绕同一套数据工作，面试时也容易解释系统如何从“AI 初稿”走到“可交付材料”。

## LLM 失败怎么办

LLM 适配层使用 OpenAI-compatible Chat Completions 接口，可以接通义千问、DeepSeek 或其他兼容模型。未配置 `LLM_API_KEY`、超时、HTTP 错误或响应解析失败时，系统会记录失败原因，并使用内置 DB2 场景化兜底结果。这样现场演示不会因为网络、额度或模型服务异常中断。

## 兜底内容是不是假数据

数据是合成数据，但不是无意义占位。内置案例覆盖 DB2 索引变更、新增字段、数据修复、REORG/RUNSTATS、锁等待应急、HADR 受控切换、表空间扩容、权限收敛、备份恢复、SQL 回放验证和分区维护，兜底输出会按场景生成不同的风险、Runbook、回滚和检查 SQL。它的目标是公开演示产品闭环，同时避免真实公司数据泄露。

## 如何体现 DBA 经验

项目没有强行连接真实 DB2，而是把 DBA 运维经验体现在变更流程里：

- 变更前要有风险评估和前置检查。
- 执行步骤要能按时间窗口落地。
- 回滚方案要写清触发条件和操作方式。
- 验收清单要关注行数、索引、访问计划、锁等待、日志使用率和应用错误。
- 交付物需要人工确认和签收，不能让 AI 直接替 DBA 拍板。

## 为什么不直接连真实 DB2

首版目标是可公开展示的 AI 产品交付能力。真实 DB2 连接会涉及网络、权限、生产数据、安全审计和脱敏，容易把项目复杂度带偏。当前版本先把变更交付流程闭环做扎实，后续可以把前置检查 SQL、监控查询和工单系统接入真实环境。

## 如何保证 AI 输出稳定

系统做了三层保障：

- Prompt 要求模型返回固定 JSON 结构。
- `normalize_response` 会补齐缺失字段，保证页面和导出不崩。
- 模型失败时使用场景化兜底，流程仍然可以完成。

测试覆盖了无 Key、真实兼容接口成功、提供方超时、结构化输出缺字段和场景化兜底。

## 如何处理审计和安全

每次模型调用都会写 `llm_call_logs`，记录 provider、model、状态、耗时和错误原因。请求/响应 payload 落库前会递归脱敏，遮蔽常见密码、Token、API Key、Authorization/Cookie/session 字段和数据库连接串口令。这样既能解释调用过程，又降低误贴敏感信息被保存在审计记录里的风险。

工单回写也做了同样的审计边界：主动发送时仍使用配置里的真实 `ITSM_WEBHOOK_URL` 和可选 Bearer Token，但写入 `work_order_writeback_logs` 和返回 API 时会遮蔽 URL 查询串里的 token/key/signature 类字段、Basic Auth 密码、Authorization/Cookie/session 字段，以及外部系统响应体或异常消息里的敏感值。这样既能保留排障证据，又不会把真实 Webhook 密钥暴露在公开演示材料里。

## 人工编辑和版本记录有什么价值

AI 结果不是最终交付。DBA 可以编辑任意交付物，保存后状态回到草稿，需要重新确认。系统会记录 AI 生成、人工编辑、人工确认三个阶段，并展示最近内容变化。这个设计能说明系统尊重 DBA 责任边界，也方便后续扩展审批流。

## 签收为什么放在 analysis_runs 上

签收对应的是“一次分析运行生成的一整套交付包”，不是单份交付物。只有 6 类交付物全部确认后才能签收。签收后如果任意交付物再次编辑，系统会重置为待签收，避免旧签收覆盖新内容。

## 导出为什么重要

数据库变更交付最终不是停留在页面上，而是要进入评审、审批、执行前沟通或归档。Markdown/PDF 导出包含封面、目录、完成度、签收状态、交付物内容、版本记录和 LLM 审计，让系统输出能变成可交付材料。

## 如何验证项目不是只能本机跑

项目提供了多层验证：

- `pytest`：验证核心业务行为。
- `python -m app.evaluation` 或 `scripts/evaluate_demo_fixtures.ps1`：验证 11 个内置 DB2 场景的交付物结构和场景关键标记。
- `scripts/final_acceptance.ps1`：跑自动化测试、离线场景评测、Alembic 迁移链和端到端冒烟。
- `scripts/release_readiness.ps1`：检查发布材料、样例包、敏感文件和运行状态。
- GitHub Actions：在 Linux runner 上安装依赖、启动服务、跑最终验收并生成样例导出。
- Dockerfile、Procfile、render.yaml、railway.json、fly.toml：覆盖容器、Render、Railway 和 Fly.io 路径。

## 当前边界和不足

当前版本刻意不做这些内容：

- 不接真实生产 DB2。
- 不做复杂权限、多租户和企业组织架构。
- 不做具体 ITSM 厂商字段映射和审批流闭环；当前提供标准回写 payload、可配置 Bearer Token Webhook、回写 attempt 日志、失败重试和审计脱敏。
- 不做多级审批流。
- 不做多模型质量评测平台；当前只保留内置 DB2 场景的离线结构和关键标记评测。

这些不是忘记做，而是首版为了一个月内交付可展示闭环而收敛范围。

## 下一步怎么扩展

优先扩展方向：

- 在现有工单导入、Webhook 回写、日志和重试 API 基础上增加具体 ITSM 字段映射、外部状态机和审批评论同步。
- 扩充真实评测样本，比较不同模型在 DBA 变更场景下的输出质量。
- 增加审批评论、多人复核和回退到历史版本。
- 对接真实 DB2 只读检查账号，自动执行前置检查 SQL。
- 引入线上观测指标，用真实复核反馈迭代提示词和场景模板。

## 面试结尾口径

这个项目的重点不是追求 DB2 全功能覆盖，而是在有限时间内交付一个可试运行、可部署、可审计、可导出的 AI 运维产品闭环。它能展示 Python Web 开发、数据库建模、AI 适配、失败兜底、审计安全、文档交付和部署验收这些能力。
